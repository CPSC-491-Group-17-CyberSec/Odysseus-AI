# ML Training Status — Pre-presentation Note

**Status as of the presentation build:** the v2 / v3 anomaly-detection models
deployed in `data/anomaly_model_v2.onnx` and `data/anomaly_model_v3.onnx` are
trained on a **synthetic-heavy** dataset:

- `data/dataset.csv` (2 000 samples) — entirely synthetic feature vectors
  generated by `scripts/generate_synthetic_dataset.py` (Gaussian distributions
  hand-tuned for "looks like benign exe" / "looks like packed malware" / etc.).
- `scripts/dataset_v3.csv` (1 239 samples) — mostly synthetic, with a small
  number of real benign files (~8 in the held-out test split). 161 of the 247
  test samples are `synthetic_malware`; the v3 model has never been evaluated
  against a representative real-malware corpus.

The headline accuracy numbers in `data/anomaly_model_v2_metadata.json` and
`data/anomaly_model_v3_metadata.json` (1.00 and 0.996 respectively) are
**accurate for the synthetic distribution they were trained on**, but they
do not reflect real-world performance. The v3 metadata's own
`score_gap = -0.136` field documents that benign and malicious score
distributions overlap on its eval split.

The v4 EMBER LightGBM model (`data/ember_lgbm_model.txt`, ~96.5 % accuracy)
*is* trained on real EMBER-2018-v2 data (600 k samples) and is the reliable
PE-detection path — it's used automatically when `features[16] > 0.5`
(isPE) and a Windows-style PE extension is present.

## Why this hasn't been retrained yet

Producing a credible v3 retrain requires:
1. A real malware corpus (MalwareBazaar, Hybrid Analysis, VX-Underground —
   all with terms-of-use considerations and triage workflow).
2. A representative benign corpus per file-category (the existing
   `scripts/collect_benign_dataset.py` is the right starting point but needs
   to harvest ~5 000+ files per category for stratified training).
3. A time-split evaluation methodology so the held-out set isn't drawn from
   the same distribution as training.
4. Re-tuning the per-file-type calibration profiles in
   `include/ai/FileTypeScoring.h` to match the new model's score distribution.

That's a multi-week effort and was deliberately scoped out of the
presentation milestone in favor of integration and stability work
(response layer wiring, EDR-Lite UI, allowlist suppression, quarantine
restore UI, cache versioning, code-signing-gated path downgrade).

## Mitigations already in place

The following changes in the current build materially reduce the impact of
the synthetic-trained v2/v3 model on demo runs:

- **`minStrongForAnomalous`** (Yazan, `FileTypeScoring.h`) — a flagging
  threshold below the `cleanCeiling` requires at least one *concrete*
  indicator, not just a high statistical score. Statistical noise alone no
  longer files findings.
- **Threshold raises across the board** (Yazan, `FileTypeScoring.h`) —
  cleanCeiling went from ~0.30 to ~0.42, anomalousCeiling 0.55 → 0.68,
  suspiciousCeiling 0.80 → 0.85. The model has less authority to flag.
- **AI-only trust downgrade** (Yazan, `FileScannerHash.cpp`) — when only
  the AI pass triggered AND the binary is `SignedTrusted`, Anomalous
  unflags entirely and Suspicious downgrades to Anomalous.
- **`pruneAIUpserted()` + `source!='AI/local'` filter** (Yazan,
  `ReputationDB.cpp`) — AI guesses are no longer self-promoted into the
  hash blocklist (closed a real cache-poisoning loop).
- **`isEmberCandidate()` extension gate** (Yazan,
  `FileScannerDetectors.cpp`) — EMBER is no longer scored against ELF /
  Mach-O / scripts that happen to have an `MZ`-looking prefix.
- **Code-signing-gated path downgrade** (this milestone,
  `FileScannerDetectors.cpp`) — `~/build/` and `~/.app/Contents/MacOS/`
  no longer get unconditional severity caps; they require positive
  trust from `CodeSigning::verifyFile()`.
- **Cache versioning** (this milestone, `ScanDatabase.cpp` +
  `CacheVersion.cpp`) — when the model files are eventually replaced,
  every cached "clean" verdict is automatically invalidated and re-evaluated
  against the new model.

## Roadmap (post-presentation)

1. Build a small but real malware corpus from MalwareBazaar (~500 PE,
   ~200 ELF, ~200 Mach-O, ~300 script samples — manageable manual triage).
2. Re-run `scripts/generate_dataset.py` against the new corpus + the
   existing `scripts/collect_benign_dataset.py` output.
3. Retrain v3 with `scripts/train_model_v3.py` (XGBoost preferred over the
   current logistic regression; both supported).
4. Re-evaluate per-type calibration in `FileTypeProfiles` and update the
   ceilings if the new score distribution shifts.
5. Bump the model file's content — the new `CacheVersion::modelVersion()`
   string changes automatically, so every cached verdict in the field is
   re-evaluated against the new model on the user's next scan.
6. Document the new accuracy numbers in
   `data/anomaly_model_v3_metadata.json` honestly, with a held-out time-split
   number alongside the cross-validation number.
